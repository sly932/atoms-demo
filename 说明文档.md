## 1. 实现思路

对标 Atoms / Lovable / v0 这类 "AI app builder" 的核心闭环，做成对话式 App 生成器：

```
登录(handle) → 描述应用 → 智能体流式生成单文件 HTML 应用 → 沙箱 iframe 实时运行
                              ├─ 多轮对话迭代(带上当前代码改)
                              ├─ 版本历史，任意回滚
                              └─ 发布 → 公开只读运行链接 /p/{id}
```

关键设计：

- **生成产物 = 单文件 HTML**（内联 CSS/JS）。让模型直接产出一个完整 `<!DOCTYPE html>` 文档，前端塞进 `iframe srcDoc` 就能跑。无需构建、无需后端执行用户代码，"真实交互" 天然成立、且安全。
- **流式生成**：服务端把模型 token 边收边转发给前端，前端累积后实时刷新 iframe，复刻 Atoms "看着它一点点长出来" 的体验。同一请求内累积完整结果，结束后落库（消息 + 版本 + 项目当前 HTML）。
- **多轮迭代**：每次追加请求时，把"当前版本的完整 HTML + 历史对话 + 新需求"一起发给模型，要求返回整份更新后的 HTML（不做 diff，简单稳定）。
- **降级兜底**：没配 key 或上游失败时，服务端返回一个确定性的、真实可交互（localStorage 持久化）的模板应用，保证端到端永远能跑通、可演示。

## 2. 关键取舍

| 维度 | 选择 | 理由 / 取舍 |
| --- | --- | --- |
| 框架 | Next.js 15 (App Router) | UI + API 一套代码；Vercel 一键部署（题目建议目标）。 |
| 模型 | DeepSeek-V4-Pro @ SiliconFlow | OpenAI 兼容 `/chat/completions` + 流式；模型/base 全 env 可配。 |
| 持久化 | **libSQL / Turso** | 既是 SQLite 语义（贴合题目"SQLite"提示），又能在 Vercel serverless（文件系统易失）上持久化。本地 `file:` 离线即可跑，线上换成 Turso 云库，**同一套客户端代码**。 |
| 预览隔离 | sandbox `iframe srcDoc` | 浏览器端运行、零服务端执行，安全且即时。 |
| 鉴权 | handle-only 签名 cookie | 满足"初始化/注册"主流程；HMAC 签名、httpOnly、防篡改、无需 session 存储。演示够用，非生产级账号安全。 |
| 迭代方式 | 每轮返回整份 HTML | 简单稳定，避免 diff 合并的脆弱性；代价是 token 开销，已列入优化项。 |

## 3. 做了哪些调研

- **Atoms / MGX 形态**：确认其本质是"描述→智能体生成应用→可视化网页运行"的 AI app builder，据此定义最小闭环。
- **SiliconFlow 接口**：确认 OpenAI 兼容 `/v1/chat/completions`、支持 `stream`，DeepSeek V4 在平台上的 model id 为 `deepseek-ai/DeepSeek-V4-Pro`。
- **Vercel 持久化**：serverless 文件系统是易失的，本地 SQLite 文件无法在线上持久化 → 选 libSQL/Turso，既保留 SQLite 体验又解决线上持久化。

## 4. 遇到的问题与解决

- **线上持久化**：见上，libSQL `file:`（本地）↔ Turso（线上）统一客户端解决。
- **流式生成同时要落库**：在同一个 `ReadableStream` 的 `start()` 里边转发 token 边在服务端累积，流结束后再写库（版本 + 消息 + 当前 HTML），保证预览实时、数据不丢。
- **模型可能输出代码围栏/前言**：`extractHtml` 兜底剥离 ```html 围栏、跳过 `<!DOCTYPE>` 前的废话；`looksLikeHtml` 校验，不合格则走降级模板。
- **同毫秒创建的版本排序不稳定**：排序加 `rowid` 兜底（`ORDER BY created_at DESC, rowid DESC`），保证版本顺序确定。该问题由数据层单测发现并修复。
- **iframe 内 localStorage**：sandbox 需 `allow-same-origin` 才能用 localStorage —— 接受这个隔离/能力的权衡（生产应改为独立沙箱域名），已在 README 标注。

**未做 / 简化**（有意控制在 6–8h）

- 生成用整份 HTML 替换，未做增量 diff（token 优化项）。
- 预览未做独立沙箱域名隔离。
- 未做多文件/框架级生成（仅单文件 HTML 应用）。

## 6. 若继续投入：扩展与优先级

1. **P0 token 与稳定性**：迭代改为"结构化 diff / 局部替换"，降低重复输出整份 HTML 的开销；生成失败重试与超时控制。
2. **P0 预览隔离**：把 iframe 预览迁到独立沙箱源（如 `*.sandbox` 子域），彻底隔离用户生成代码。
3. **P1 真实账号**：密码 / OAuth + 项目协作（共享、只读/可编辑）。
4. **P1 生成能力升级**：多文件项目、引入组件/数据（表单→存储）、可选模型与温度。
5. **P2 产品化**：模板库 / 公开作品广场（remix 他人应用）、生成过程可视化（plan→code→preview 分步）、导出代码 zip。
